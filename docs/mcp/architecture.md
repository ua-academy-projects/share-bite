# Share Bite MCP Architecture Standards

This document defines shared implementation standards for all Share Bite MCP servers.
It applies to every server under `mcp/`, including guest, business, admin-auth, and future MCP integrations.

## Goals

- Expose service capabilities through MCP without bypassing existing Go APIs.
- Keep MCP servers thin, auditable adapters over HTTP APIs.
- Use consistent configuration, authentication, error, pagination, request ID, and timeout patterns.
- Make side effects explicit so LLM clients can reason safely about tool calls.

## Server Boundaries

MCP servers must not connect directly to PostgreSQL, Redis, S3, queues, or other raw infrastructure. They must call the owning Go service API and rely on that service for authorization, validation, and business rules.

| MCP server | Owning Go API | Primary purpose |
| --- | --- | --- |
| `mcp/guest-server` | Guest API | Guest/customer read and action workflows |
| `mcp/business-server` | Business API | Business owner profile, venue, analytics, and recommendation workflows |
| `mcp/admin-auth-server` | Admin/Auth API | Administrative and moderation workflows with audit logging |

## Transport Modes

### stdio

Use stdio for local desktop clients and development. The MCP process is started by the client and communicates over standard input/output.

Required properties:

- Logs must go to stderr or files, never stdout.
- Secrets may be provided by environment variables for local use.
- Fallback auth tokens are allowed only for local development and must never be returned by tools/resources.

### Streamable HTTP

Use Streamable HTTP for deployed MCP servers and remote clients.

Required properties:

- Auth must be forwarded from request headers or a trusted gateway.
- Request IDs must be accepted from the caller or generated by the MCP server.
- HTTP deployments should use network-level rate limits in front of the MCP process.

## Standard Configuration Naming

All MCP servers should use these names for MCP runtime settings:

| Setting | Environment variable | Required | Default |
| --- | --- | --- | --- |
| Transport | `MCP_TRANSPORT` | No | `stdio` |
| Host | `MCP_HOST` | No | `127.0.0.1` |
| Port | `MCP_PORT` | No | server-specific |
| Path | `MCP_PATH` | No | `/mcp` |
| Request timeout | `<SERVICE>_REQUEST_TIMEOUT_SECONDS` | No | `10` |
| API base URL | `<SERVICE>_API_BASE_URL` | Yes | none |
| Fallback auth token | `<SERVICE>_API_AUTH_TOKEN` | No | none |

Examples:

- `GUEST_API_BASE_URL`
- `GUEST_API_REQUEST_TIMEOUT_SECONDS`
- `BUSINESS_API_BASE_URL`
- `BUSINESS_API_REQUEST_TIMEOUT_SECONDS`
- `ADMIN_AUTH_API_BASE_URL`
- `ADMIN_AUTH_API_REQUEST_TIMEOUT_SECONDS`

Legacy environment variables may be supported temporarily, but new code should use the standard names.

## Tool Categories

Tools must fit one of these categories:

| Category | Meaning | Required metadata |
| --- | --- | --- |
| Read-only | Fetches data and has no side effects | Description states read-only behavior |
| Write | Creates, updates, deletes, triggers, or marks state | Required permissions and side effects |
| Admin write | Administrative mutation or privileged operation | Required permissions, side effects, and audit logging |
| Confirmation-required | High-impact or irreversible action | Confirmation token or explicit confirmation parameter |

## Tool Description Requirements

Every tool description must clearly state:

- whether the tool is read-only or has side effects;
- required roles or permissions;
- what Go API/service it calls;
- whether it forwards the caller's auth token;
- whether it emits audit logs;
- pagination behavior, if applicable;
- confirmation requirements, if applicable.

## Pagination Pattern

List tools must accept `limit` and either `offset` or `cursor`. Current Share Bite APIs mostly use offset pagination.

Standard defaults:

- `limit`: default `20`;
- `max_limit`: `100`;
- `offset`: default `0`;
- negative offsets clamp to `0`;
- invalid limits clamp into `[1, max_limit]`.

Tool responses should include:

```json
{
  "items": [],
  "total": 0,
  "pagination": {"limit": 20, "offset": 0}
}
```

## Request IDs

Every outbound Go API request must include `X-Request-ID`.

- If the MCP caller provides a request ID, forward it.
- Otherwise generate a UUID.
- Return request IDs only as metadata or non-secret response fields.

## Timeout And Retry Policy

- Default timeout: 10 seconds.
- Use a single standard timeout setting per MCP server.
- Do not retry non-idempotent write calls unless the Go API explicitly supports idempotency keys.
- Read-only calls may retry transient connection failures once when the operation is safe.
- Tool responses should map timeouts into structured MCP-safe errors.

## Shared Package

Shared helpers live in `mcp/sharebite_common`.

The package provides reusable patterns for:

- auth token extraction and normalization;
- secret redaction;
- error mapping;
- pagination clamping;
- confirmation checks;
- audit event shape;
- request ID generation;
- test fakes and security assertions.
